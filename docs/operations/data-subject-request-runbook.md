# 정보주체 권리 요청 처리 Runbook

> 개인정보 보호법 제35조 (열람), 제36조 (정정/삭제), 제37조 (처리정지) 기준.

## 1. 요청 접수 경로

- 앱 내: 설정 > 회원 탈퇴 (즉시 처리)
- 앱 내: 설정 > 동의 내역 관리 (즉시 처리)
- 이메일: privacy_officer_email (HD-01 확정)
- ADMIN: `dashboard/users/rights` 페이지 (운영자 수동 처리)

## 2. SLA

- 즉시 처리 가능 항목: **즉시** (앱 내 자가 처리)
- 운영자 검토 필요 항목: **10일 이내** (보호법 권고)
- 복잡한 경우: **+ 20일 연장 가능** (사용자 통지 필수)

## 3. 처리 절차

### 3.1 본인 확인

- 앱 내: 비밀번호 재입력 + (선택) TOTP
- 이메일: 이메일 본문에 회원 정보 + (필요 시) 추가 인증
- ADMIN: 운영자가 본인 확인 후 처리

### 3.2 요청 유형 별 처리

#### 열람 요청 (제35조)

1. `adminPipaRoutes /api/v1/admin/users/:id/pipa/export` 호출
2. 사용자 데이터 export (JSON / CSV)
3. 이메일로 안전한 링크 전송 (24시간 만료)
4. `audit_logs` 기록

#### 정정 요청 (제36조 제1항)

1. 운영자 확인 → ADMIN 에서 `users` 테이블 수정
2. 변경 사유 5자 이상 기록
3. `audit_logs` 자동 기록
4. 사용자에게 처리 결과 통지

#### 삭제 요청 (제36조 제2항)

1. 사용자 본인 확인
2. `adminPipaRoutes /api/v1/admin/users/:id/pipa/finalize` 호출
3. 즉시 익명화 (deletion-policy.md 의 5번 참조)
4. **법령상 의무 보관 데이터** (audit_logs, 동의 이력) 는 비식별화 후 유지
5. deletion certificate 발급
6. 사용자에게 이메일 통지

#### 처리 정지 요청 (제37조)

1. 사용자 본인 확인
2. `users.account_status = 'paused'`
3. 로그인 차단 + 모든 데이터 처리 정지
4. 사용자가 재활성 요청 시 복구

#### 이전 요청 (제35조의2)

1. 열람 요청과 동일 절차 + 사용자가 지정한 다른 사업자로 이전
2. 이전 받는 사업자의 보안 수준 검증

## 4. 처리 거부 사유 (법령 제35조 제5항)

다음 경우 처리 거부 가능 (사용자에게 사유 통지):

- 법령 의무 보관 데이터 (audit_logs 5년)
- 진행 중인 분쟁/소송 관련 데이터
- 다른 사용자의 권리 침해 우려 (예: 피드 게시물 삭제 시 댓글 작성자 의도)

## 5. 운영자 점검

- ADMIN `dashboard/users/rights` 에서 큐 모니터링
- 미처리 요청 > 7일 → secops 알람
- 분기 1회 요청·처리 통계 보고 (DPO 에게)
